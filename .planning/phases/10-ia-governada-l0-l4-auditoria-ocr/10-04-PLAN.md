  - `queryAuditLogs(filters: AuditFilters): Promise<{ success; rows?: AuditLogRow[]; error? }>`:
    - getActor (auth → users.id/tenant_id/role).
    - ROLE GATE: `if (!AUDIT_PERMITTED_ROLES.includes(actor.role)) return { success:false, error:'Acesso restrito' }`. This gate is the access-control boundary BECAUSE the next step uses the admin client.
    - Use `createAdminClient()` — DEFINITIVE, not createClient(). Rationale (verified): the v1 `audit_logs_tenant_select` RLS policy restricts SELECT to `get_my_role() IN ('admin','superadmin')`; auditor/dpo are NOT in that allowlist, so a createClient() RLS read returns ZERO rows for them and breaks AUD-03. The admin client bypasses RLS, so tenant isolation is enforced MANUALLY by an explicit `.eq('tenant_id', actor.tenant_id)` (never trusted from the client). No conditional/fallback — admin client + role gate + explicit tenant filter is the single path.
    - Build the query: `.from('audit_logs').select('id, actor_id, action, table_name, record_id, old_values, new_values, created_at').eq('tenant_id', actor.tenant_id)`; conditionally `.eq('table_name', filters.tableName)`, `.eq('actor_id', filters.actorId)`, `.gte('created_at', dateFrom)`, `.lte('created_at', dateTo)`; `.order('created_at', { ascending: false })`; `.range(offset, offset + AUDIT_PAGE_SIZE - 1)` where offset = (page ?? 0) * AUDIT_PAGE_SIZE.
    - Return rows (old_values/new_values included for the before/after diff).
