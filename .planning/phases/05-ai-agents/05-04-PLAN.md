GET (Meta hub verification):
- Read hub.mode / hub.verify_token / hub.challenge from query. If `mode==='subscribe'` AND
  `token === process.env.WHATSAPP_WEBHOOK_VERIFY_TOKEN` → `new Response(challenge, { status: 200 })`,
  else 403. (Call-time env read.)

POST (inbound):
- `const rawBody = await request.text()` FIRST (Pitfall 3 — before any JSON.parse).
- `const sig = request.headers.get('x-hub-signature-256')`. If
  `!verifyWhatsAppSignature(rawBody, sig, process.env.WHATSAPP_APP_SECRET ?? '')` → 403. (Reject unsigned.)
- `const payload = JSON.parse(rawBody)`. Extract first message + wamid (entry[0].changes[0].value.messages[0]).
  If no message (e.g. status-only event) → return 200 immediately (nothing to do).
- Dedup: `createAdminClient()` insert into `whatsapp_inbound_events { wamid, from_phone, payload }`. On 23505
  (duplicate) → return 200 (idempotent skip — Pitfall 8). Only proceed past the dedup insert on success.
- Return 200 IMMEDIATELY; fire-and-forget `processInbound(payload, admin).catch(console.error)`.

processInbound (async):
- Determine intent + appointmentId:
  - button reply (type 'button' → button.payload, OR type 'interactive' → interactive.button_reply.id):
    `buttonPayloadToStatus(payload)` → { appointmentId, status }.
  - text (type 'text' → text.body): `classifyConfirmationIntent(body)` → confirm/cancel/ambiguous; map
    confirm→'confirmado', cancel→'cancelado', ambiguous→NO update. For text path, resolve the appointmentId
    via the patient's most recent next-day pending confirmation outreach (look up agent_outreach_log by
    patient phone/from_phone OR the latest appointment for that phone in the confirmation window). If no
    appointment can be resolved → treat as ambiguous (no status change).
- On confirmado/cancelado with a resolved appointmentId: derive tenant from the appointment row (NOT payload),
  `admin.from('appointments').update({ status }).eq('id', appointmentId)`; write agent_outreach_log row
  { agent_type:'confirmation', appointment_id, status:'responded', intent_result, whatsapp_message_id: wamid };
  `logBusinessEvent({ tenantId, actorId:null, action:'ai02.confirmation.reply', details })`.
- On ambiguous: write agent_outreach_log { agent_type:'confirmation', status:'ambiguous', intent_result:'ambiguous',
  whatsapp_message_id: wamid } and DO NOT touch appointments (human review). logBusinessEvent.
- Mark whatsapp_inbound_events.processed = true at the end.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Meta→/api/webhooks/whatsapp | Untrusted inbound HTTP from the public internet |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-webhook-S | Spoofing | forged inbound POST | mitigate | HMAC-SHA256 X-Hub-Signature-256 over RAW body with WHATSAPP_APP_SECRET + timingSafeEqual; reject unsigned (403) before any processing |
| T-5-webhook-verify | Spoofing | fake GET subscription | mitigate | GET requires `hub.verify_token === WHATSAPP_WEBHOOK_VERIFY_TOKEN` else 403 |
| T-5-webhook-T | Tampering | replayed inbound → double status update | mitigate | `whatsapp_inbound_events.wamid` UNIQUE; insert-before-process, 23505 → idempotent skip |
| T-5-webhook-I | Information Disclosure / Integrity | tenant taken from payload | mitigate | tenant derived from the matched appointment row, never from payload |
| T-5-intent | Tampering | LLM misclassifies free text → wrong status | mitigate | safe fallback: ambiguous (or unresolved appointment) → NO status change, logged 'ambiguous' for human review; LLM returns 'ambiguous' when key absent |
| T-5-secret-wa | Information Disclosure | WHATSAPP_APP_SECRET / verify token leak | mitigate | server-only, call-time read, never NEXT_PUBLIC_ |
</threat_model>
