## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client form → createTransaction (Server Action) | User-supplied account_id/cost_center_id could reference another tenant's account/CC. |
| client form → cadastro Server Actions | Non-admin could attempt writes; cross-tenant parent_id/account_id could be injected. |
| Asaas (external) → webhook insert | Auto-post path must resolve defaults without trusting external payload for classification. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-09 | T (Tampering) | createTransaction account_id/cost_center_id | mitigate | RLS WITH CHECK on financial_transactions + the referenced FK rows are RLS-isolated by clinic_id; a cross-tenant id is invisible to the authenticated client and the INSERT FK resolves only within tenant. Server Action also re-derives tenant_id from actor, never from input |
| T-14-10 | E (Elevation) | cadastro writes by non-admin | mitigate | every create/update gates `actor.role IN ('admin','superadmin')` before insert; RLS admin_write policy is the backstop |
| T-14-11 | T (Tampering) | updateCategoryAccount cross-tenant accountId | mitigate | action fetches the account and confirms `clinic_id = actor.tenant_id` + leaf + matching type before UPDATE; RLS blocks reading other tenants' accounts anyway |
| T-14-12 | T (Tampering) | chart cycle / depth abuse | mitigate | createAccount rejects parent depth >= 2 and relies on (clinic_id,code) unique to prevent dupes; ON DELETE RESTRICT prevents orphaning |
| T-14-13 | D (DoS) / availability | webhook blocked by CC lookup | mitigate | default-CC lookup is best-effort; failures leave cost_center_id null and the handler still returns 200 (D-03b) |
</threat_model>
