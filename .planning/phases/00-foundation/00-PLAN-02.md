<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated user → PostgreSQL (PostgREST) | Every row access crosses RLS; tenant isolation enforced here |
| Supabase Auth → token issuance (hook) | Hook runs with elevated rights to inject claims; must read only the source-of-truth users table |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-06 | Denial of Service (C-1) | users RLS policy | mitigate | Policy uses `get_my_tenant_id()` SECURITY DEFINER (bypasses RLS internally) instead of `SELECT FROM users`; verify script rejects any `FROM users` in the policy file. Prevents infinite recursion / stack-depth lockout. |
| T-00-07 | Elevation of Privilege (C-5) | custom_access_token_hook | mitigate | Hook reads tenant_id + role from `public.users` (service-role-writable only), never from user-mutable `user_metadata`. Verify script rejects `user_metadata`. |
| T-00-08 | Tampering (Pitfall 5) | tenants/users write policies | mitigate | Every write policy has both USING and WITH CHECK, blocking cross-tenant inserts/updates. |
| T-00-09 | Repudiation / Tampering (SEC-02) | audit_logs table | mitigate | RLS denies DELETE and UPDATE (`USING (false)`); no INSERT policy (writes only via future SECURITY DEFINER triggers). Immutable compliance log. |
| T-00-10 | Information Disclosure | search_path injection (Pitfall 7) | mitigate | Both SECURITY DEFINER functions set `SET search_path = public`. |
| T-00-11 | Information Disclosure | audit_logs read scope | mitigate | SELECT restricted to same tenant AND role in (admin, superadmin) via JWT claim. |
</threat_model>
