<objective>
Deliver the member-entry lifecycle (D-04, D-05, D-09, D-10): admin invites staff by email (Resend, FYNXIA-branded) OR creates them directly with a temporary password; invitations expire in 24h, single-use, re-invite invalidates the prior; an invited user accepts via `/invite/[token]`, sets a password, and is linked to the clinic with the assigned role. Patient infrastructure (no UI) reuses the same mechanism: recepcionista invite (role=patient) and a public `POST /api/invitations` endpoint for `/agendar` self-registration. Every invite event is audited via `logBusinessEvent` (SEC-02 manual business audit, D-13).

Purpose: Completes ROADMAP success criterion #2 (admin invites dentist → dentist logs in → tenant-isolated) and lays the patient-account backend for Phase 2. Email branding is FYNXIA-only (D-16); Supabase's native invite email is suppressed in favor of Resend.

Output: invitation Server Actions, two react-email templates, the Resend wrapper, the public invite accept page, the patient self-register API route, an admin team page with the invite form.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin form → invitation Server Action | invite/create input crosses into privileged server path |
| public /invite/[token] → acceptInvitation | unauthenticated user redeems a token to gain an account |
| public POST /api/invitations → DB | unauthenticated patient self-register request |
| invitation action → audit_logs | business-event write path (SECURITY DEFINER / service role) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-15 | Elevation of Privilege | non-admin creates invitations to add users | mitigate | invitations_admin_write RLS (get_my_role IN admin/superadmin) + server-side role check in createInvitation |
| T-01-16 | Tampering | expired/reused invite token redeemed | mitigate | acceptInvitation checks status='pending' AND expires_at >= now(); flips expired to status='expired'; single-use (status→accepted) |
| T-01-17 | Spoofing | invite assigns elevated role to attacker | mitigate | role taken from server-stored invitation row, never from client input at acceptance |
| T-01-18 | Denial of Service | public /api/invitations abused to mass-create rows | accept | low-value (no auth account created; recepcionista confirms in Phase 2); rate-limiting deferred to Phase 3 security headers/edge |
| T-01-19 | Repudiation | invite/create action without audit evidence | mitigate | logBusinessEvent on every path (INVITE_SENT/ACCEPTED/REVOKED, USER_CREATED_DIRECT) — SEC-02 |
| T-01-20 | Information Disclosure | invite token leaks clinic/email in URL | mitigate | token is gen_random_uuid (unguessable); /invite page reveals only clinic name + role, email read-only |
</threat_model>
